# Park302 대시보드 생성 프롬프트 초안 (1차 정리본)

이 문서는 `Park302` 프로젝트의 1차 기획안이다.  
현재 단계에서는 **프로젝트 초기 세팅, 공통 레이아웃 구성, 사이드바 메뉴 구조, 기본 라우팅 설계**를 우선 목표로 하며, 상세 기능 구현은 확정된 요구사항을 기준으로 순차 진행한다.

이 문서는 Claude에게 바로 코드 구현을 요청하기 위한 최종 구현 명세서가 아니라,  
**초기 기획 및 구조 설계를 위한 합의 문서**로 사용한다.

따라서 Claude에게 요청할 때는 아래 원칙을 따른다.

1. 먼저 이 문서를 기준으로 누락 사항, 모호한 표현, 기술적 리스크를 검토한다.
2. 바로 기능 구현으로 넘어가지 않고, 레이아웃과 구조 설계를 우선 진행한다.
3. 1차 범위로 확정된 내용만 구현 대상으로 삼는다.
4. 2차 이후 기능은 설계 검토 후 순차적으로 확장한다.

# 1. 기획 초안

## 1-1. 왜 만드는지

### 프로젝트 개요
- **서비스명:** `Park302`
- **내부 프로젝트명:** `park302-dashboard`
  - JAR 명, GitHub Repository 명 등 내부 식별자는 `park302-dashboard`로 통일한다.

### 한 줄 정의
- `Park302`는 내가 관리하는 업체와 프로젝트의 **문의, 공지, 인프라, 정산, 작업 현황**을 한 곳에서 통합 관리하는 **1인용 운영 대시보드**이다.

### 프로젝트 구현 의도
- 하나의 대시보드에서 업체별 문의사항과 운영 정보를 일관되게 관리할 필요가 있다.
- 기존에는 카카오톡, 유선전화, 메일 등 여러 채널로 문의와 요청이 분산되어 있어, 히스토리 추적과 우선순위 관리에 많은 비용이 들었다.
- 따라서 문의, 공지, 진행 상태, 인프라, 정산, 운영성 데이터 등을 한 화면 구조 안에서 조회하고 관리할 수 있는 전용 관리자 시스템을 구축하고자 한다.

### 1차 목표
- 업체별 **문의, 공지, 진행 상태, 인프라, 정산 현황**을 하나의 화면 구조에서 조회하고 관리할 수 있도록 한다.

---

## 1-2. 누가 쓰는지

### 대상 사용자
- 본인 1인 전용 관리자 시스템으로 사용한다.
- 외부 고객이 직접 로그인하는 기능은 현재 범위에 포함하지 않는다.
- 일반 사용자용 화면은 제공하지 않는다.
- 현재는 단일 관리자만 사용하며, 다중 관리자 확장은 1차 범위에서 제외하고 추후 검토한다.
- 1인 전용 시스템이더라도 **로그인, 인증, 인가**는 필수로 적용한다.

### 권한 범위
- 로그인한 `admin` 사용자만 모든 메뉴에 접근할 수 있으며, 기본적으로 읽기/쓰기가 가능하다.
- 단, 외부 DB 수정 기능과 같은 민감 작업은 일반 CRUD보다 더 강한 확인 절차를 적용한다.
- 주요 변경 작업은 반드시 로그를 남긴다.

## 1-3. 어떤 문제를 해결하는지

### 문제 정의
- 문의가 카카오톡, 전화, 메일 등 여러 경로로 들어와 일관되게 관리하기 어렵다.
- 문의 기록과 응답 이력이 분산되어 있어 중앙 관리가 필요하다.
- 업체별 운영 정보가 흩어져 있어 한 곳에서 통합 조회하기 어렵다.
- 공지 전달이 유선 또는 수동 방식에 의존하고 있어 비효율적이다.
- 업체별 메뉴 등 운영성 데이터를 수작업으로 관리해야 하는 경우가 많다.
- 유지보수, 추가개발, 진행 업무를 일정과 리스트 관점에서 함께 보기 어렵다.

### 해결 방향
- 문의와 응답 이력을 중앙 시스템에서 관리한다.
- 업체별 운영 정보를 통합 조회할 수 있도록 한다.
- 공지를 ERP 또는 시스템 단위로 전달할 수 있는 구조를 마련한다.
- 운영성 테이블(`menu` 등) 관리 효율을 높일 수 있는 기반을 만든다.
- 작업 현황을 캘린더와 리스트 형태로 함께 볼 수 있도록 한다.

## 1-4. 어떤 기능이 필요한지

### 핵심 업무 시나리오 1: 고객사 ERP 내 문의 기능
- ERP를 구축한 각 업체의 우측 상단에 `개발팀문의` 메뉴를 제공한다.
- 사용자가 해당 메뉴를 클릭하면 **모달 형태의 문의 UI**가 열리고, 문의 유형별(문의/추가개발/오류 등) 목록을 조회하거나 새 문의를 등록할 수 있어야 한다.
- 문의 데이터는 `Park302` 프로젝트의 DB에서 중앙 관리한다.
- 고객사 ERP는 `Park302` API를 호출하여 업체별 문의 목록 조회 및 글/댓글의 등록, 수정, 삭제를 처리한다.
- 각 ERP는 동일한 UI 및 동일한 연동 방식을 사용한다.
- 고객사 측 문의 화면은 별도 페이지가 아니라 공통 모달 UI 형태로 제공한다.

### 핵심 업무 시나리오 2: 대시보드 내 문의 통합 관리
- `Park302`에서는 업체별 문의글을 리스트 형태로 통합 관리할 수 있어야 한다.
- 문의에는 댓글 및 대댓글 기능이 있어야 하며, 이를 통해 고객사와 소통할 수 있어야 한다.
- 내가 `Park302`에서 댓글로 응답하면 고객사 ERP에서도 동일 내용을 확인할 수 있어야 한다.
- 고객이 문의글을 작성하거나 댓글을 남기면, 나에게 **내부 알림**이 우선 표시되고 이후 Slack 알림도 연동할 수 있도록 한다.
- 읽음/미읽음 상태를 관리한다.

### 문의/댓글 데이터 관리 방식
- 문의 및 댓글의 원본 데이터는 모두 `Park302`에 저장한다.
- 고객사 ERP는 `Park302` API를 통해 데이터를 조회하고 등록/수정/삭제한다.
- 고객사 ERP에서 문의를 등록하면 `Park302` API를 즉시 호출하는 방식으로 처리한다.
- 화면의 실시간 자동 갱신까지는 포함하지 않으며, 새로고침 또는 재조회 시 반영되는 **준실시간 구조**를 전제로 한다.
- 댓글/대댓글은 대시보드 프로젝트의 `work` 테이블을 참조하는 `reply` 테이블을 통해 관리한다.

### 문의 기능의 단계별 범위
- **1차 범위**
  - `Park302` 내부의 문의관리 화면 구조, 목록 UI, 상태 설계
- **2차 범위**
  - 고객사 ERP와의 댓글 연동
  - 내부 알림 및 Slack 알림 연동

### 핵심 업무 시나리오 3: 업체 정보 및 인프라 관리
- `업체관리` 메뉴를 통해 내가 구축한 업체의 운영 정보를 조회하고 관리할 수 있어야 한다.
- 업체 목록은 리스트 형태로 표시하며, 등록/수정/상세 조회는 모달을 통해 처리한다.

### 업체관리 대상 데이터
1. 기본정보
   - 회사명, 대표자명, 주소, 전화번호, 휴대폰번호, 팩스번호, ERP URL, 홈페이지 URL 등
2. 담당자 정보
   - 담당자명, 연락처, 이메일, 부서 등
3. ERP 사양
4. 서버/DB/인프라 정보
5. API 키 및 연동 정보
6. 계약/정산 정보
7. 메모 및 특이사항
8. 개발 진행 상황

### 핵심 업무 시나리오 4: 외부 DB의 `menu` 테이블 관리
- 업체관리 메뉴에서 외부 업체 DB의 `menu` 테이블을 조회할 수 있는 구조를 마련한다.
- `menu` 테이블은 메뉴 추가/삭제, depth, 정렬순서 등의 관리 대상이 될 수 있다.
- 표시 UI는 TUI Grid 기반으로 구성한다.

### 외부 DB `menu` 관리 범위
- **1차 범위**
  - 업체별 `menu` 테이블의 조회 구조 설계
  - 읽기 전용 기반의 화면 구조와 데이터 접근 방식 정의
- **2차 범위**
  - 수정 기능(추가/삭제/정렬/깊이 변경) 검토 및 확장

### 외부 DB 수정 기능 전제 조건
- 외부 DB 접속 정보는 암호화 저장한다.
- 수정 전 사용자 확인 절차를 거친다.
- 변경 이력을 저장한다.
- 오류 발생 시 롤백 또는 복구 전략을 마련한다.
- 공통 스키마 여부를 사전에 확인한다.

### 핵심 업무 시나리오 5: 공지사항 등록 및 알림
- `공지관리` 메뉴를 통해 특정 업체를 선택하여 공지사항을 등록할 수 있어야 한다.
- 공지는 인프라 장애, 점검, 업데이트 예정 사항 등을 ERP 차원에서 전달하기 위한 목적을 가진다.

### 공지 기능의 단계별 범위
- **1차**
  - `Park302`에서 공지 등록, 조회, 대상 업체 지정
- **2차**
  - 고객사 ERP 또는 기타 알림 채널로 실제 전파

### 공지 대상 및 알림 기준
- 각 업체의 공지 대상 직원은 해당 업체 DB의 `member` 테이블에서 `agent_id = 0`인 직원으로 식별한다.
- 공지 알림은 ERP 내부 알림을 기본으로 한다.
- 공지사항은 읽음 여부를 별도로 관리하지 않는다.

### 핵심 업무 시나리오 6: 메인 대시보드 + 캘린더
- 메인 페이지는 대시보드 역할을 수행한다.
- 캘린더를 중심으로 내가 등록한 작업 리스트와 업체의 유지보수/추가개발 요청을 표시할 수 있어야 한다.
- 대시보드 상단 또는 하단에는 상태별 요약 카드 또는 통계 영역을 배치한다.
- 문의 건수 및 상태(완료/보류/취소/진행중/대기 등)를 구분하여 표시한다.
- 캘린더와 리스트 보기 전환 기능을 제공하며, 기본 화면은 캘린더로 한다.

## 1-5. 무엇을 1차 범위로 할지

### 기능 우선순위 / MVP 범위
- 1차 구현 범위는 아래로 한정한다.
  - 프로젝트 생성
  - 라이브러리 설치
  - 환경설정
  - 공통 레이아웃
  - 사이드바 메뉴
  - 기본 라우팅
  - 빈 페이지 골격

- 초기 단계에서는 **사이드바 메뉴와 레이아웃 구조를 우선 세팅**하고, 본문 콘텐츠는 최소한의 정적 구조만 배치한다.

### 대시보드
- 메인 페이지가 대시보드 역할을 수행한다.
- `FullCalendar React`를 사용하여 정적 샘플 캘린더 UI를 우선 배치한다.
- 실제 API 연동 및 일정 데이터 연결은 추후 설계한다.
- `리스트보기` 전환 버튼은 UI 구조만 우선 제공한다.

### 업체관리
- 1차에서는 `업체관리` 메뉴와 빈 목록 페이지 레이아웃까지만 구성한다.
- 실제 컬럼, 필터, 상세 데이터 연동은 추후 진행한다.

### 문의관리
- 1차에서는 `문의관리` 메뉴와 목록형 화면 구조를 우선 정의한다.
- 데이터 모델은 추후 확정한다.
- `work` 테이블은 가칭이며, 실제 명칭은 추후 확정한다.

### 공지관리
- 1차에서는 메뉴와 화면 골격을 우선 구성한다.
- 목록, 등록, 수정 UI가 필요하다.
- 등록/수정은 모달을 통해 처리한다.
- 대상 업체 다중 선택 UI가 필요하다.
- 본문 작성용 에디터가 필요하며, 이미지 첨부 가능성도 고려한다.

### 데이터 관점의 엔티티 정의
- 상세 엔티티 설계는 환경설정 및 레이아웃 구조가 먼저 잡힌 후 진행한다.
- 현재 단계의 엔티티 초안은 아래와 같다.
  - `agent` : 업체
  - `work` : 문의사항
  - `reply` : 댓글
  - `file_info` : 내부 첨부파일 및 정적 리소스 관리
  - `env` : 환경설정
  - `board` : 기타 게시판
  - `notice` : 공지/알림
  - `log` : 이력

### 화면 목록과 메뉴 구조
- 레이아웃 및 메뉴 구성은 업로드한 레퍼런스 이미지를 참고한다.
- 기본 메뉴 구조 초안은 아래와 같다.
  - 대시보드
    - 캘린더
    - 리스트
    - 상세조회(모달)
  - 업체관리
    - 리스트
    - 등록/수정/조회(모달)
  - 문의관리
    - 리스트
    - 등록/수정/조회(모달)
  - 공지관리
    - 리스트
    - 등록/수정/조회(모달)
  - 게시판
    - 리스트
    - 등록/수정/조회(모달)
  - 설정관리
    - 리스트
    - 등록/수정/조회(모달)

# 2. 기술 / 아키텍처 제약

## 2-1. Spring Boot + React

### 백엔드 / 프론트 구조
- Language: `Java 17`
- Build Tool: `Gradle`
- Backend Framework: `Spring Boot 3.x` 계열의 검증된 안정 버전 사용
- Security: `Spring Security`
- ORM / Query
  - 기본 데이터 접근은 `Spring Data JPA` 기반으로 구성한다.
  - 복잡한 조회가 필요할 경우 `Querydsl` 도입을 추후 검토한다.

### 프론트엔드 구조
- 프론트엔드는 `React SPA`로 구성한다.

- `Thymeleaf` 등 서버 템플릿 엔진은 사용하지 않는다.
- 백엔드는 `Controller - Service - Repository` 계층 구조를 유지한다.
- 프론트엔드는 별도 React 애플리케이션으로 구성한다.

## 3-3. 코드 스타일
- 유지보수성을 최우선으로 고려한다.

- 지나치게 축약된 로직이나 과한 공통화는 지양한다.
- React/JavaScript 코드에서는 과도한 한 줄 체이닝, 복잡한 삼항연산, 불필요한 추상화를 지양한다.
- 공통 함수와 모듈은 `utils`, `lib`, `hooks`, `services` 등 역할 기준으로 분리한다.
- 공통 스타일은 `common.css` 또는 `styles` 폴더에서 관리한다.
- 공통 함수 중 아래 코드 활용하되, 수정할 부분이 있으면 제안해주고, 확정되면 수정
    ```javascript
    /**
    * fetch 래퍼
    * - JSON Content-Type 기본 설정 (multipart 전송 시 headers에서 제외할 것)
    * - HTTP 에러(response.ok=false) 시 Error throw
    * - 응답 JSON 자동 파싱 (204 No Content는 null 반환)
    *
    * @param {string} url
    * @param {RequestInit} options
    * @returns {Promise<any>}
    */
    export const gFetch = async (url, options = {}) => {
    const { headers, ...rest } = options;

    const res = await fetch(url, {
        headers: {
        'Content-Type': 'application/json',
        ...headers,
        },
        ...rest,
    });

    if (!res.ok) {
        const err = new Error(`HTTP ${res.status}`);
        err.status = res.status;
        try {
        err.data = await res.json();
        } catch {
        err.data = null;
        }
        throw err;
    }

    if (res.status === 204) return null;
    return res.json();
    };

    /* ─────────────────────────────────────────────
    * UI — Toast (입력 검증·간단 안내, 우측 상단)
    * ───────────────────────────────────────────── */

    /**
    * 우측 상단 토스트 (입력값 검증 오류·간단 안내용)
    * - react-toastify 기반, ToastContainer가 앱에 마운트되어 있어야 함
    *
    * @param {string} msg
    * @param {'info'|'success'|'error'|'warning'} type
    */
    export const gToast = (msg, type = 'info') => toast[type](msg);

    /* ─────────────────────────────────────────────
    * UI — 로딩
    * ───────────────────────────────────────────── */

    /**
    * 로딩 다이얼로그 표시 (gLoadingMessage 역할)
    * - adminCloseLoading() 으로 닫을 것
    *
    * @param {string} title
    */
    export const gLoading = (title = '처리 중...') => {
    Swal.fire({
        title,
        allowOutsideClick: false,
        showConfirmButton: false,
        didOpen: () => Swal.showLoading(),
    });
    };

    /**
    * 로딩 다이얼로그 닫기
    */
    export const gCloseLoading = () => Swal.close();

    /* ─────────────────────────────────────────────
    * UI — 성공 메시지
    * ───────────────────────────────────────────── */

    /**
    * 성공 메시지 (중앙, 확인 버튼 + 3초 타이머 자동 닫힘, gMessage 역할)
    *
    * @param {string} title  - 굵은 제목 (예: "저장 완료")
    * @param {string} html   - 본문 내용 (예: "회사정보가 저장되었습니다.")
    * @returns {Promise}
    */
    export const gSuccess = (title, html = '') =>
    Swal.fire({
        title,
        html: html || undefined,
        icon: 'success',
        timer: 3000,
        timerProgressBar: true,
        showConfirmButton: true,
        confirmButtonText: '확인',
    });

    /* ─────────────────────────────────────────────
    * UI — 오류/안내 메시지
    * ───────────────────────────────────────────── */

    /**
    * 오류·안내 메시지 (중앙, 확인 클릭 시 닫힘, gMessage alert 역할)
    *
    * @param {string} title  - 굵은 제목 (예: "저장 실패")
    * @param {string} html   - 본문 내용 (예: "저장 중 오류가 발생했습니다.")
    * @param {'error'|'warning'|'info'} icon
    * @returns {Promise}
    */
    export const gAlert = (title, html = '', icon = 'error') =>
    Swal.fire({
        title,
        html: html || undefined,
        icon,
        confirmButtonText: '확인',
    });

    /* ─────────────────────────────────────────────
    * UI — Confirm
    * ───────────────────────────────────────────── */

    /**
    * Confirm 다이얼로그 (중앙, gMessage confirm 역할)
    * - 확인 → true, 취소/외부클릭 → false
    * - Promise<boolean> 반환 (await 사용 가능)
    *
    * @param {string} title  - 굵은 제목 (예: "회사정보 저장")
    * @param {string} html   - 본문 내용 (예: "저장하시겠습니까?")
    * @returns {Promise<boolean>}
    */
    export const gConfirm = (title, html = '') =>
    Swal.fire({
        title,
        html: html || undefined,
        icon: 'question',
        showCancelButton: true,
        confirmButtonText: '확인',
        cancelButtonText: '취소',
    }).then((r) => r.isConfirmed);
    ```

- 상태 전이 로직에는 주석을 충분히 작성한다.

## 3-5. Alert / Confirm UX 규칙
- `SweetAlert2`는 confirm, alert, question, deny 등 모달형 사용자 피드백에 사용한다.
- `React-Toastify`는 가벼운 알림과 입력 검증 안내에 사용한다.

### React-Toastify 사용 예
- 폼 검증 메시지
- 간단한 안내 메시지
- 저장 성공/실패의 가벼운 피드백

### Confirm 적용 원칙
- 삭제 작업은 기본적으로 confirm을 적용한다.
- 외부 전파가 있는 작업은 confirm을 적용한다.
- 대량 변경 작업은 confirm을 적용한다.
- 외부 DB 수정과 같은 민감 작업은 confirm을 적용한다.
- 일반적인 등록/수정은 상황에 따라 confirm 적용 여부를 결정한다.
